# memory-agentcore v0.1.1: Bug Fixes, API Alignment & Core Features

## Summary

Fix 7 bugs, align plugin API interfaces with OpenClaw conventions, and add 3 core features: `agentcore_share` tool, `enabled` toggle, and startup validation.

## Reference

API patterns verified against the [openclaw-vertexai-memorybank](https://github.com/Shubhamsaboo/openclaw-vertexai-memorybank) plugin (110 stars, `kind: "general"`, same plugin type as ours). Source verified by reading the full `index.ts` from GitHub on 2026-03-19.

## Scope

- **In scope**: Bug fixes, API interface alignment, share tool, enabled config, startup validation
- **Out of scope**: Self-improvement tools, markdown mirror, hybrid scoring, offline sync queue (v2)

## Part 1: API Interface Alignment

These changes align our plugin API usage with the patterns verified in the `openclaw-vertexai-memorybank` reference (same `kind: "general"` plugin).

### 1.1 Config access: `api.pluginConfig`

**Current** (incorrect):
```typescript
configSchema: {
  parse(value: unknown) {
    const raw = ... ;
    return resolveConfig(process.env, raw);
  },
},
register(api) {
  const config = this.configSchema.parse(api.config) as PluginConfig;
```

**Change to**:
```typescript
register(api: any) {
  const raw = (api.pluginConfig ?? {}) as Record<string, unknown>;
  const config = resolveConfig(process.env, raw);
```

**How it works**: OpenClaw reads `openclaw.plugin.json`'s `configSchema` to validate user config at load time. The validated config is passed to the plugin via `api.pluginConfig`. We keep `openclaw.plugin.json` for schema validation (and the Control UI), but use `api.pluginConfig` at runtime. Our `resolveConfig()` applies env var overrides on top of the validated config.

Remove the `configSchema` property from the plugin object (OpenClaw uses `openclaw.plugin.json` directly). Keep the `resolveConfig()` function for env var precedence.

### 1.2 Workspace directory: `api.workspaceDir`

**Current**: `api.resolveAgentDir()`
**Change to**: `api.workspaceDir || process.env.OPENCLAW_WORKSPACE || defaultPath`

### 1.3 Tool registration: direct object, not factory

**Current**:
```typescript
api.registerTool(() => createRecallTool(client!, config));
```

**Change to**:
```typescript
api.registerTool({
  name: "agentcore_recall",
  label: "AgentCore Recall",
  description: "...",
  parameters: { ... },
  async execute(_toolCallId: string, params: Record<string, unknown>) { ... }
});
```

All 7 existing tools + the new share tool will be registered as direct objects inside `register()`. The separate `tools/*.ts` factory files become helper functions that return the tool config object (no behavioral change, just registration pattern).

### 1.4 Tool return format: `{ content, details }`

**Current**: Returns bare objects like `{ results: [...], count: 5 }`

**Change to**:
```typescript
return {
  content: [{ type: "text" as const, text: JSON.stringify({ results, count }, null, 2) }],
  details: { count: results.length },
};
```

All 8 tools (7 existing + share) adopt this format.

### 1.5 `agent_end` hook: check `event.success`

**Current**: No success check
**Change to**:
```typescript
api.on("agent_end", async (event: any) => {
  if (!event.success) return;  // Skip capture on agent errors/cancellations
  // ... rest of capture logic
});
```

The `event.success` field (boolean) is `false` when the agent run ended with an error or was cancelled. Capturing failed runs would store noise (error messages, incomplete reasoning). Verified from vertexai-memorybank source: it checks `if (!event.success) return;` before any capture.

### 1.6 CLI registration: add commands list

**Current**: `api.registerCli((ctx) => { ... })`
**Change to**:
```typescript
api.registerCli(
  (ctx: any) => { ... },
  { commands: ["agentcore-status", "agentcore-search", "agentcore-list", "agentcore-forget", "agentcore-episodes", "agentcore-stats", "agentcore-sync"] }
);
```

### 1.7 AWS SDK timeout: nested under `requestHandler`

**Current**: `timeoutMs` config exists but is never used
**Change to**:
```typescript
this.client = new BedrockAgentCoreClient({
  region: config.awsRegion,
  credentials: ...,
  maxAttempts: config.maxRetries,
  requestHandler: {
    requestTimeout: config.timeoutMs,
  },
});
```

## Part 2: Bug Fixes

### 2.1 Remove `client!` non-null assertions

**Problem**: ~15 uses of `client!` in index.ts. If client initialization fails, runtime crashes.

**Fix**: Introduce a `ready` flag. Tools and hooks check it and return graceful errors:

```typescript
let client: AgentCoreClient | null = null;
let ready = false;

// In tools:
if (!client || !ready) {
  return {
    content: [{ type: "text", text: "AgentCore plugin not ready. Check config and connection." }],
    details: { error: "not_ready" },
  };
}

// In hooks:
if (!client || !ready) return;

// CLI commands always work (for debugging):
if (!client) { console.error("Client not initialized."); return; }
```

### 2.2 Auto-recall: parallelize namespace queries

**Problem**: Sequential `for...of` loop querying N namespaces.

**Fix**:
```typescript
const results = await Promise.allSettled(
  namespaces.map(ns =>
    client.retrieveMemoryRecords({ query: prompt, namespace: ns, topK: config.autoRecallTopK })
  )
);
const allRecords = results
  .filter((r): r is PromiseFulfilledResult<MemoryRecordResult[]> => r.status === "fulfilled")
  .flatMap(r => r.value);
```

### 2.3 Search tool: return `nextToken` and accept `next_token`

**Add parameter**:
```typescript
next_token: { type: "string", description: "Pagination token from previous search" }
```

**Add to return**: `nextToken: result.nextToken || undefined`

### 2.4 Episodes tool: remove `include_reflections`

Remove from `parameters.properties` and from `execute` body. No behavioral change since it was never implemented.

### 2.5 File-sync: log in catch blocks

```typescript
} catch (err) {
  this.log.debug(`[file-sync] Cannot read directory ${fullDir}: ${err}`);
}
```

### 2.6 Namespace sanitize

**Rationale**: Namespace paths are user-influenced strings (from scope config, session keys, agent IDs). AWS AgentCore uses these as namespace identifiers. Characters like `/`, `..`, or special characters could cause unintended namespace traversal or API errors.

New function in `scopes.ts`:
```typescript
function sanitizeId(input: string): string {
  return input.replace(/[^a-zA-Z0-9_\-\.]/g, "_");
}
```

Allows: alphanumeric, underscore, hyphen, dot (common in agent IDs like `sales-bot.v2`).
Blocks: `/`, `..`, spaces, special characters.

Applied in `scopeToNamespace()` for the dynamic `id` portion only (static prefixes like `/agents/` are safe):
```typescript
case "agent": return `/agents/${sanitizeId(scope.id!)}`;
```

And in `buildEpisodicNamespace()` for actorId/sessionId.

## Part 3: New Features

### 3.1 `agentcore_share` tool

New file: `src/tools/share.ts`

```typescript
name: "agentcore_share"
label: "AgentCore Share"
description: "Share a memory across multiple scopes/namespaces. Creates the same record in each target namespace."
parameters: {
  content: string (required) — the memory to share
  target_scopes: string[] (required) — e.g. ["agent:sales-bot", "project:ecommerce"]
  category?: string — preference/fact/decision/entity/other
  importance?: number — 0.0-1.0
  tags?: string[]
}
```

Logic:
1. Validate: if `target_scopes` is empty or missing, return error
2. For each target scope, resolve namespace via `scopeToNamespace(parseScope(scope))`
3. For each namespace, call `client.batchCreateRecords()` (one record per namespace)
4. Metadata includes `source: "shared"`, `sharedAt: ISO timestamp`
5. Return `{ content: [{type: "text", text: JSON.stringify({shared, targetNamespaces, recordIds})}], details }`

Edge cases:
- Empty `target_scopes`: return `{ shared: false, error: "target_scopes is required and must not be empty" }`
- Malformed scope string (e.g. `"agent"` without `:id`): `parseScope()` returns `global`, which is valid. This is acceptable — the caller gets records in `/global` namespace
- Partial failure: if some namespaces succeed and others fail, return partial results with `{ shared: true, failed: [...] }`

### 3.2 `enabled` config toggle

Add to `PluginConfig`:
```typescript
enabled: boolean;  // default: true, env: AGENTCORE_ENABLED
```

In `register()`, check first:
```typescript
if (!config.enabled) {
  api.logger.info("[agentcore] Plugin disabled via config");
  return;
}
```

Add to `openclaw.plugin.json` configSchema.

### 3.3 Startup validation with `ready` flag

In `registerService.start()`:
```typescript
async start() {
  try {
    await client!.listMemoryRecords({ namespace: "/global", maxResults: 1 });
    ready = true;
    api.logger.info(`[agentcore] Service started, connection verified`);
  } catch (err) {
    ready = false;
    api.logger.error(`[agentcore] Startup validation failed: ${err}. Tools/hooks disabled, CLI still available.`);
  }
}
```

Tools and hooks check `ready` flag. CLI commands always work for debugging.

**Timing**: OpenClaw calls `service.start()` during gateway startup, before any agent runs. So `ready` is set before any tool/hook invocations. No race condition.

**Recovery**: If the initial validation fails (e.g., temporary network issue), the user can restart the gateway to retry. Future enhancement: add a retry mechanism in `start()`.
